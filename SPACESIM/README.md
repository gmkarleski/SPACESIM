## Planned prototype scope

Per `../docs/NETCODE_CONTRACT.md` §10 (validation criteria), the prototype implements and validates the sim-tick boundary architecture by running the cube-launches-to-moon-landing scenario:

1. Cube launches from a planet surface.
2. Cube reaches orbit using thrust and physics.
3. Cube transfers to a moon.
4. Cube captures into moon orbit.
5. Cube lands on the moon surface.
6. Time-warp works during the transfer.
7. No graphics required — math correctness is the bar.

The validation milestone is binary: the cube either completes the launch-to-landing sequence with correct authoritative-state advancement at 30 Hz fixed sim-tick, or it does not. When the cube lands successfully under time-warp through the transfer, the prototype validates the contract and Phase 0 closes.

The cube scenario tests the following contract components:

- **30 Hz fixed sim-tick.** Authoritative state advances at 30 Hz; rendering is decoupled.
- **Mode transitions.** Cube launches in PhysX-active mode, transitions to Kepler-rails after escaping the atmosphere, and back to PhysX-active for the moon descent.
- **Floating origin at 50 km.** The cube crosses the 50 km threshold during launch and during landing approach; the floating-origin shift must work without disturbing the cube's trajectory.
- **Save format.** Save / quit / reload mid-transfer; the cube continues from the saved state.
- **Authoritative-state-only determinism.** Outside PhysX-active mode (Kepler-rails during the transfer), the simulation is deterministic; inside PhysX-active mode (launch and landing), simulation is non-deterministic by design.

## First-open instructions (user verification step)

The file-level setup from commit 027 is complete. To verify the setup works:

1. Open Unity Hub. Open the SPACESIM project.
2. Wait for Unity to import packages (`com.unity.mathematics` and `com.unity.burst` are new; first open will resolve them). The console may report package resolution activity; this is expected.
3. From the Unity menu, select **Prototype → Verify Setup**. Confirm the console logs:
   - Unity version 6000.4.7f1
   - Several `package:` lines, including `com.unity.mathematics` and `com.unity.burst`
   - The line `Editor scripting infrastructure verified`
4. In the Project window, open `Assets/Scenes/TestFoundation.unity`. The scene is empty.
5. Create an empty GameObject in the scene (right-click in Hierarchy → Create Empty). Name it "PrototypeTest".
6. With "PrototypeTest" selected, in the Inspector click **Add Component**. Search for `PrototypeStartupTest` and attach it.
7. Save the scene (Ctrl+S / Cmd+S).
8. Press **Play**. Confirm the console logs:
   - `Phase 0 prototype foundation ready`
   - `Unity version: 6000.4.7f1`
   - A platform identifier and a persistent data path.
9. Stop play mode. The smoke test passes if all three lines appeared without errors.

After step 8 succeeds, the file-level setup is verified at runtime. The next commit (commit 028) will add the `.meta` files Unity generated for the new assets to git, plus a commit recording the verification pass.
