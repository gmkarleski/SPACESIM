# 046: SOI crossing predictor

Land the second concrete event predictor in the Phase 1 event-prediction infrastructure that commit 045 stood up. `SoiCrossingPredictor` analyzes a Kepler-rails vessel's orbit and predicts the earliest tick at which the vessel will cross an SOI boundary — either outward exit from the current body's SOI or inward entry into one of its children's SOIs. `VesselEventPredictionDriver` is extended to call the new predictor alongside `PeriapsisApoapsisPredictor`, populates `KeplerState.NextSoiTransitionTick`, and enqueues `SimEventType.SoiCrossing` entries on the priority queue.

The commit lands in three stages with verification gates between them:

- **Stage 1:** `SoiCrossingPredictor` static class with outward closed-form (conic equation) + inward sampled-and-refined (coarse-sample + bisection) math paths. 14 EditMode tests covering both paths, earliest-wins across children, near-tangent graze, horizon clamp, and defensive null inputs.
- **Stage 2:** Driver integration in `VesselEventPredictionDriver.PredictAndUpdate` with per-predictor try/catch isolation. Populates `NextSoiTransitionTick` and enqueues `SoiCrossing` events. 4 EditMode integration tests.
- **Stage 3:** NETCODE_CONTRACT §2.3 amendment (bidirectional semantics + null-cases extension), KeplerState/SimEventType XML doc cleanup, PHASE_TRACKER + DECISIONS updates, this artifact.

The predictor sits in the extensibility hook CONSTRAINTS §2 locked at commit 045: write a `*Predictor` static class, add an enum value to `SimEventType` (already there as `SoiCrossing` since commit 045), populate the corresponding `KeplerState` field (`NextSoiTransitionTick` was scaffolded), call `EventQueue.UpdateVesselEntry`. No architectural change.

## Architecture overview

The SOI crossing predictor breaks into three cooperating pieces, each living at the right layer:

**1. Pure-function math (Stage 1).** `SoiCrossingPredictor.PredictNextCrossing` is a pure function — no state, no caching, no side effects, no Unity dependencies beyond `Unity.Mathematics.math` for the double3 vector ops. It reads `currentBody.PositionWorld`, `currentBody.SoiRadiusMeters`, `currentBody.Mu`, `currentBody.ParentBody` (via `BodyRegistry.GetChildrenOf(currentBody)` for the child enumeration), and per-child reads `child.PositionWorld` + `child.SoiRadiusMeters`. None of these mutate; the predictor produces one `long?` output. This matches CONSTRAINTS §2's locked design ("Event predictors are pure functions of orbital and trajectory state. No hidden state, no caches that can drift").

The two math paths reflect the genuinely different geometric problems they solve. Outward: vessel exits a sphere centered at its own gravitational focus — this is a conic-equation-on-the-orbit problem, closed-form solvable for the constant-body-position case. Inward: vessel enters a sphere centered NOT at the focus — this is a vessel-position-vs-sphere-NOT-at-focus problem with no general closed-form analog in ν, so sampled+refined is the right shape. The asymmetry isn't an inconsistency; it's the math.

**2. Driver wiring (Stage 2).** `VesselEventPredictionDriver.PredictAndUpdate` calls the new predictor in the same per-vessel context where it already calls `PeriapsisApoapsisPredictor.Predict`. Both predictors get their own try/catch — per-predictor isolation rather than per-vessel-all-or-nothing. If one throws, the other's writes still land. The KeplerState field assignment (`kepler.NextSoiTransitionTick = soiCrossingTick`) and the queue update (`EventQueue.UpdateVesselEntry(vesselId, SimEventType.SoiCrossing, soiCrossingTick)`) both handle null cleanly: assignment to a `long?` field accepts null; `UpdateVesselEntry(..., null)` removes any existing entry. Stale-value cleanup is automatic.

**3. Contract amendments (Stage 3).** NETCODE_CONTRACT §2.3 already had `next_soi_transition_tick` typed as `Option<SimTickCount>` from commit 045's nullable-fields amendment. Stage 3 just sharpens the semantic prose (bidirectional, predictor-populated, null-cases enumerated). The schema didn't change; only the documentation.

The signature divergence between `SoiCrossingPredictor.PredictNextCrossing(state, body, ...)` and `PeriapsisApoapsisPredictor.Predict(state, currentTick, mu, ...)` deserves explicit comment because it could look like inconsistency at a glance. It isn't: the SOI predictor needs `currentBody` for child enumeration and position math, so taking the body and reading `body.Mu` internally is the cleaner shape (single source of truth, one fewer parameter). The PeriapsisApoapsis predictor doesn't need the body at all — its math is intrinsic to orbit shape — so it takes `mu` directly. Different inputs because they solve different problems; the asymmetry is documented in both predictors' XML docs.

## Design decisions

See `docs/DECISIONS.md` "SOI crossing predictor (commit 046)" for the full multi-part design (9 sub-decisions a-i), 6 alternatives rejected, and rationale. The headline decisions:

- **Two math paths** (outward closed-form via conic equation, inward sampled-and-refined via bisection).
- **Constant-body-position assumption** acceptable for Phase 0/1; documented loudly for Phase 4+ revisit.
- **Lookahead horizon:** one orbital period for elliptical, ~1 game year for hyperbolic.
- **Coarse sample interval = 100 ticks** (3.33 sec at 30 Hz). Pragmatic detection mode misses near-tangent grazes inside one coarse sample window; the driver's at-evaluation-time check is the safety net.
- **Earliest-crossing-wins across all children** in the predictor — differs from `VesselSoiRerootingDriver`'s first-match-wins semantics; both are correct in their respective contexts.
- **`DetectionAggressiveness` enum in the signature**, Pragmatic hardcoded at v1 call sites; Strict reserved for Phase 5/6 UI toggle.
- **`BoundaryHugTolerance` = 1.0m** for the numerical near-circular-at-SOI edge case.
- **Signature divergence** from `PeriapsisApoapsisPredictor` (takes body, not mu) is math-driven, not stylistic.
- **Per-predictor try/catch isolation** in the driver: if SoiCrossing throws, PeriapsisApoapsis updates still land and vice versa.

## Lessons

**Test geometry needs analytic verification before committing.** Initial test setups for tests 5/7/8 placed children where the vessel's orbital path didn't visit — for example, the orbit's apoapsis-end was at -X but the child was at +X, so the vessel never approached the child SOI. The geometry was caught by walking through the orbit math during writing (`ArgumentOfPeriapsis = 0` and `TrueAnomalyAtEpoch = 0` puts periapsis at +X, apoapsis at -X; sweeping ν through 0 → π takes the vessel through +Y, not back through +X). Lesson: when a test setup involves orbital geometry, trace the vessel's path analytically before committing — don't trust the "obvious" placement.

**Newton-Raphson Kepler solver fails to converge at e > ~0.9 with the 15-iteration limit.** The first round of tests had elliptical orbits at e ≈ 0.96 (rPeri = LEO, rApo = Moon-distance scale). The solver returned garbage anomalies which propagated into the predictor as garbage tick estimates. Stage 1 round 2 raised periapsis to 2e8 m across all multi-million-km tests, keeping `e < 0.8` for solver stability. Each elliptical test now has an in-test `Assert.Less(e, 0.8, ...)` sanity guard. Lesson: when calling solver-dependent math, constrain input ranges to the solver's stable region. If Phase 5/6 gameplay demands higher-e transfer orbits (e.g., highly eccentric capture orbits, Hohmann to slow co-rotating moons), swap Newton-Raphson for Laguerre's method (cubic convergence; same iteration-loop structure; Conway 1986 covers it as the Newton-Raphson alternative). Markley 1995 ("Kepler equation solver" in Celestial Mechanics & Dynamical Astronomy, 63:101-111) is a separate speed optimization — closed-form initial guess + one Newton iteration — that doesn't fix high-e but speeds up the common case by ~10x.

**Crossing-tick semantics: predictor returns the tick when vessel ENTERS the SOI sphere, not the tick when vessel reaches child center.** Test 7 (`MultipleChildSois_ReturnsEarliest`) initially asserted the crossing tick should be within one coarse-sample of the vessel-at-NearChild-center tick. The actual predictor (correctly) returns the tick when distance equals SOI radius — which is earlier by the SOI-radius-traversal time at vessel speed (~1.13M ticks earlier on the test orbit). The test assertion was loosened to two looser conditions: (a) crossing fires BEFORE vessel reaches NearChild center, (b) crossing is closer in absolute distance to NearChild's center-arrival than to FarChild's. The second assertion is the actual earliest-wins discriminator and is robust to the SOI-radius traversal offset. Lesson: when testing predictors that return event times, the "event" is the boundary crossing, not the closest approach.

**Per-predictor try/catch isolation has no clean throw-injection test path** (same pattern as commit 045 Stage 2). All the obvious failure modes for `SoiCrossingPredictor.PredictNextCrossing` (null state, null currentBody, infinite SOI, no children) return null gracefully rather than throwing. Making the predictor throw deterministically would require external mocking infrastructure (e.g., a body subclass that overrides `Mu` to throw). The isolation discipline is structural in the driver code and documentary in the XML doc; commit 045's `EventPredictionDriver_VesselThatThrows_LoopContinues` test established the precedent of acknowledging this gap rather than mocking around it.

**`Unity.Mathematics.math` doesn't have `long` overloads for `abs` or `min`.** Subtraction of two `long` ticks produces `long`, which won't match `math.min(int, int)` or `math.abs(int)` overloads. Either silent coercion to double (precision loss near `long.MaxValue`) or compile error depending on overload resolution. Used `System.Math.Min(long, long)` and `System.Math.Abs(long)` defensively. Lesson: when arithmetic on `long`s feeds into a math library, verify the library has long overloads or fall back to `System.Math`.
