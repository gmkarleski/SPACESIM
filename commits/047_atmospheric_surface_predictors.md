# 047: Atmospheric entry + surface impact predictors

Land the third and fourth concrete event predictors in the Phase 1 event-prediction infrastructure. `AtmosphericEntryPredictor` and `SurfaceImpactPredictor` analyze a Kepler-rails vessel's orbit and predict the earliest tick at which the vessel crosses the atmospheric outer boundary or the body's surface radius respectively. Both are thin wrappers around a shared math helper (`OrbitalElements.SolveConicAtRadius`) extracted from commit 046's outward closed-form path. The driver aggregates both predictions into `KeplerState.NextModeTransitionTick` via min-of-both. Three predictors now populate the event queue.

The commit lands in three stages with verification gates between them:

- **Stage 1:** `ReferenceBody` schema additions (`SurfaceRadiusMeters` + `AtmosphericTopAltitudeMeters`), `OrbitalElements.SolveConicAtRadius` helper extraction, `OrbitalElements.BoundaryHugTolerance` migration to a shared public const, `SoiCrossingPredictor` refactored to delegate to the new helper (behavior-preserving). 8 EditMode tests for the helper.
- **Stage 2:** `AtmosphericEntryPredictor` and `SurfaceImpactPredictor` static classes as thin wrappers; integrated into `VesselEventPredictionDriver.PredictAndUpdate` with per-predictor try/catch isolation; `NextModeTransitionTick` aggregation via `MinNullable`. 16 EditMode tests across both predictors and driver integration.
- **Stage 3:** NETCODE_CONTRACT §2.3 + §2.7 amendments (next_mode_transition_tick semantics, BodyState surface/atmosphere fields), stale-XML-doc cleanup (KeplerState, SimEventType, Vessel.cs, TransitionEvaluation.cs), PHASE_TRACKER + DECISIONS updates, this artifact.

The two new predictors share their math with `SoiCrossingPredictor`'s outward closed-form path via the extracted helper. Three callers now share one conic-equation solve. The helper-extraction pattern follows "extract on the second or third caller, not the first" — extracting on commit 046's single caller would have been premature.

## Architecture overview

Commit 047 fits within the predictor-extensibility hook established by commit 045 and matured by commit 046. Three layers of work, each at the right abstraction:

**1. Schema (Stage 1).** `ReferenceBody` gains two scalar fields — surface radius and atmospheric-top altitude. Both are body-intrinsic properties that the math needs to know about; both are scalar (not curves or functions) because the event predictors only need the boundary altitude, not the full density-vs-altitude profile. The Phase 5 atmospheric flight model will need the full profile (NETCODE_CONTRACT §2.7 lists `atmospheric_profile` as Phase 4+ deferred), but that's a separate concern from event prediction. Keeping the schema minimal at this stage avoids prejudging the Phase 5 model's shape.

**2. Math (Stage 1).** The conic equation `r(ν) = p / (1 + e·cos(ν))` solved for ν given a threshold radius is one piece of math. Commit 046 inlined it inside `SoiCrossingPredictor.TryPredictOutwardCrossing` because there was no shared usage yet. Commit 047 introduces two more callers needing the same solve with different thresholds. Extracting the math to `OrbitalElements.SolveConicAtRadius` at commit 047 (second + third caller arriving simultaneously) is the right time. The helper is body-agnostic (takes a scalar `targetRadius`, not a `ReferenceBody`); body-specific logic — infinite-SOI early returns, vacuum-body checks, child enumeration — lives one layer up in each caller. `BoundaryHugTolerance` migrates alongside because it's a shared math concern.

**3. Driver (Stage 2).** `VesselEventPredictionDriver.PredictAndUpdate` already has per-predictor try/catch isolation from commit 046. Commit 047 adds two more inner try/catch blocks following the same pattern, with one extra step: an aggregation of the two new ticks into `KeplerState.NextModeTransitionTick` via `MinNullable`. The aggregation step lives outside the per-predictor try/catches so it always runs — if either predictor threw, that local variable stays null and the min handles null gracefully. The `MinNullable` private helper duplicates `SoiCrossingPredictor.MinNullable` (used there for earliest-wins across children) — intentional locality, not over-engineering.

The signature divergence between the new predictors and `SoiCrossingPredictor` deserves the same explicit framing as the commit-046 divergence from `PeriapsisApoapsisPredictor`: math-driven, not stylistic. `SoiCrossingPredictor` has the `DetectionAggressiveness` parameter because its inward path is sampled-and-refined with a granularity knob. `AtmosphericEntryPredictor` and `SurfaceImpactPredictor` have one math path (closed-form conic solve via the shared helper); no granularity to tune, no parameter needed. Different inputs because they solve different problems. Both predictors' XML docs make this explicit.

The deferred `TransitionTriggerReason` rename is the one piece of friction commit 047 doesn't fully resolve. The trigger evaluator (`Vessel.IsAtmosphericEntryPredicted`) reads `NextModeTransitionTick` and fires `TransitionTriggerReason.AtmosphericEntryPredicted` regardless of which event populated the field. Post-commit-047, this label is technically imprecise — surface impact populating `NextModeTransitionTick` would fire the trigger with an atmospheric-entry-labeled reason. Renaming the enum value or splitting it into separate reasons is mechanical (touches the enum, one callsite in `Vessel.cs`, and tests) but conflates with the predictor-implementation scope. Deferring keeps commit 047 focused; the imprecision is documented in three XML docs (the enum value, `IsAtmosphericEntryPredicted`, `EvaluateTransitionTriggers`) and in the DECISIONS entry so future readers know it's intentional and known.

## Design decisions

See `docs/DECISIONS.md` "Atmospheric entry + surface impact predictors (commit 047)" for the full multi-part design (10 sub-decisions a-j), 5 alternatives rejected, and rationale. The headline decisions:

- **Two `ReferenceBody` schema fields**: `SurfaceRadiusMeters` (default 6.371e6 Earth-like), `AtmosphericTopAltitudeMeters` (default 0 vacuum).
- **`<= 0` for vacuum body**: defends against both 0.0 default and accidentally-negative inputs.
- **Threshold radii**: atmospheric entry = surface + atmospheric top; surface impact = surface. Both delegate to the shared helper.
- **Shared math helper `OrbitalElements.SolveConicAtRadius`** extracted from commit 046's outward path. Three callers now share the conic-equation solve.
- **`BoundaryHugTolerance` migrated** to `OrbitalElements` as public const (shared with the new helper).
- **`NextModeTransitionTick` aggregates via `MinNullable`** of atmospheric + surface ticks. Extensible to N-way for future predictors.
- **No `DetectionAggressiveness` on new predictors**: closed-form math has no aggressiveness behavior to differ. Signature asymmetry with `SoiCrossingPredictor` is math-driven.
- **Atmospheric entry semantics: next threshold crossing**, gameplay-relevant for vessels above the atmosphere; benign over-detection for vessels already inside.
- **Test SetUp reflection-sets `surfaceRadiusMeters = 1.0`** (point-mass default) so existing tests aren't broken by the new schema field. New convention for future schema additions.
- **`TransitionTriggerReason.AtmosphericEntryPredicted` label is imprecise post-aggregation**; rename deferred to a separate cleanup commit. Three XML docs flag the imprecision.

## Lessons

**Helper extraction post-second-caller.** When commit 046 wrote `SoiCrossingPredictor`'s outward closed-form, the math was inline. There was no shared usage yet, so extracting to a helper would have been premature abstraction — a one-call-site helper is just indirection without payoff. Commit 047 introduced two more callers (atmospheric entry, surface impact) needing the same math with different thresholds, and that was the moment to extract `OrbitalElements.SolveConicAtRadius`. The pattern is "extract on the second or third caller, not the first." Extracting on the first caller speculates about future shared usage; extracting on the fifth caller has paid duplication cost across four call sites. The second-or-third-caller window is where the abstraction is justified by evidence (you've seen the pattern repeat) but the duplication cost hasn't compounded.

**BuildState helper signature mismatch surfaced as CS1739 compile errors during Stage 2.** Multiple `BuildState` helpers exist across test files (one per file: `OrbitalElementsTests`, `SoiCrossingPredictorTests`, `AtmosphericEntryPredictorTests`, `SurfaceImpactPredictorTests`). I initially wrote the new ones with short parameter names (`a`, `e`, `trueAnomalyAtEpoch`) while the established files use long names (`semiMajorAxis`, `eccentricity`, `trueAnomalyAtEpoch`). Named-argument call sites in the new files attempted to pass `eccentricity: 0.0` to a helper that took `e` — compile error. Fix was straightforward (rename the helper parameters to match the established convention + update the two named-arg sites). The deeper observation: when introducing new test files alongside established ones, signature conventions need to match without explicit prompting. Future test infrastructure improvement could consolidate the four `BuildState` helpers into one shared utility, but each file's helper also bakes in test-specific scaffolding (e.g., `_body.BodyId` defaults), so consolidation isn't free. Deferred as a recurring test-infra concern; noted for a future cleanup pass.

**`TransitionTriggerReason` imprecision: a single field aggregating multiple event types creates downstream label ambiguity.** When commit 043 designed `TransitionTriggerReason.AtmosphericEntryPredicted`, atmospheric entry was the only contributor to `NextModeTransitionTick`. Post-commit-047 with min-aggregation across atmospheric + surface (and future scheduled-burn + interstellar-arrival), the enum value's name no longer matches what actually fires the trigger. The fix is mechanical (rename or split the enum value) but out of scope for commit 047. The architectural lesson: when adding a new event type to an aggregating field, audit downstream label consumers for new imprecision. The audit is cheap to do at design time; doing it post-implementation (as here) requires either fixing it (scope creep) or documenting the imprecision (what commit 047 did, via three XML doc updates and a DECISIONS entry). Pattern: aggregating fields outgrow the labels chosen at their introduction; expect this and plan a cleanup pass when the second contributor arrives.

**Test SetUp default mutation as schema-evolution defense.** Adding new schema fields to a class (here `SurfaceRadiusMeters` on `ReferenceBody`) can break existing tests by changing the production-default behavior. With the default `6.371e6` surface radius and existing tests using LEO-scale orbits with mild eccentricity (rPeri = 6.3e6), the new `SurfaceImpactPredictor` would have fired surface-impact events on orbits that were "fine" pre-commit-047. The fix was to reflection-set `_body.surfaceRadiusMeters = 1.0` in `VesselTests.cs` SetUp, creating a point-mass default for tests that don't care about surface behavior. The SetUp comment documents this convention so future test writers see immediately that tests exercising surface-impact behavior must reflection-set finite values per-test. Pattern: when adding a new schema field with a non-zero default that affects downstream behavior, set the field to a test-friendly value (zero / negligible / sentinel) in test SetUp. Future schema additions on `ReferenceBody` should follow the same pattern — the SetUp comment is the forward-looking memo.
