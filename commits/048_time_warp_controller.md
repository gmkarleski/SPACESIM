# 048: Time-warp controller

Time-warp rate machinery with rational rate representation, event-halting per vessel classification, PhysX cap policy, and end-to-end UI for verification. The commit replaces the pre-existing `SimTickWarpController` (float-multiplier 1× hard cap) with a singleton `WarpController` MonoBehaviour exposing a rational `WarpRate` representation, mode-aware ceilings (5×/10000×/100000× per `PhysicsMode`), event-bus halt surfacing (`OnRateChanged`, `OnWarpHalted`), and a scene-wired debug UI in `TestVessels.unity`. The atmospheric / surface predictor pair from commit 047 gets wired through to halt registration via a routine-supply gate on `IVessel`. End-to-end time-warp now works in the prototype — pause, resume, all seven discrete rates (1×/5×/10×/100×/1000×/10000×/100000×), continuous slider, clear-halt, all live in Play mode against a running sim.

The commit lands in five stages with verification gates between them:

- **Stage 1 (f9200ac) — Schema and infrastructure.** `WarpRate` rational struct + factories. `IsRoutineSupply` field added to `Vessel` (Inspector-visible boolean). `TransitionTriggerReason` enum fully split: atmospheric entry and surface impact each get their own trigger reason; `WarpRateForcedRails` added as held infrastructure for a future force-transition commit. `KeplerState.NextModeTransitionTick` split into per-predictor fields (`NextAtmosphericEntryTick` + `NextSurfaceImpactTick`). NETCODE_CONTRACT + ARCHITECTURE + DECISIONS amended in lockstep.
- **Stage 2 (8c3654a) — `WarpController`.** Singleton `MonoBehaviour` with rational rate machinery, mode-aware ceiling, event-bus halt surfacing. `WarpHaltInfo` struct + `WarpHaltReason` enum land in the SimTick asmdef as a parallel taxonomy to `TransitionTriggerReason` (preserves the Vessels-references-SimTick asmdef direction). `SimTickController` integrates via null-safe `ComputeAnalyticIterations`. The pre-existing `SimTickWarpController` is replaced wholesale: 4 file deletions, 17 obsolete tests removed, 25 new tests added (22 `WarpControllerTests` + 3 `SimTickController` integration tests).
- **Stage 3 (b7b8d3a) — Predictor halt registration.** `IVessel` exposes `IsRoutineSupply` (FakeVessel POCO updated). `VesselEventPredictionDriver` registers halt events at the imminent-tick window (`predicted <= currentTick + 1`). Gating rules: atmospheric entry and surface impact halt unconditionally; SOI crossing halts only for non-routine vessels. 10 new tests across all gating paths.
- **Stage 4 (8ce6e0f) — UI scaffolding.** `WarpUIController` MonoBehaviour with 13 `[SerializeField]` references. `TestVessels.unity` Canvas wired with pause/resume buttons, seven discrete rate buttons, continuous slider, clear-halt button. TMP Essentials imported (first TMP usage in codebase — sets the precedent for future UI work). MonoBehaviour lifecycle refactor mid-stage: singleton check moved `Awake` → `Start` to eliminate a race with `WarpController.Instance`; `OnEnable`/`OnDisable` event-subscription pair replaced with `Start`/`OnDestroy` for the debug-UI lifecycle. End-to-end verified in Play mode.
- **Stage 5 (this commit) — Tests + documentation.** 7 new tests for determinism and forbidden-state gating filling the gaps recon identified in `WarpControllerTests.cs`. `DECISIONS.md` Stage 1 entry amended to reflect the actual cap-and-stay policy (the original sketch was force-transition; Stage 2 reshape replaced it). This artifact.

## Architecture overview

Commit 048 sits at the intersection of three subsystems: the sim-tick spine from commit 033, the predictor infrastructure from commits 045–047, and the (newly-introduced) UI surface for Phase 1. Five layers of work, each at the right abstraction:

**1. Rate representation (`WarpRate`).** Float multipliers accumulate drift over long sessions. A 10,000× rate held for hours of real time produces millions of integer-tick advancements; under repeated float multiplication the accumulator slowly diverges from the exact value. `WarpRate` solves this at the type level: rational `(numerator, denominator)` integer arithmetic is exact under repeated application. The `AdvanceTicks(realTime, pendingNumerator)` helper carries forward fractional-tick remainder for future fractional modes (v1 denominator is always 1, so the helper stays exercised only by `WarpController` and the test suite's exact-arithmetic determinism tests).

**2. Mode-aware ceiling (`WarpController.EffectiveRate`).** The effective rate is always `min(CurrentRate, CeilingFor(ActiveVesselMode))`. PhysXActive vessels cap at 5× because PhysX accuracy degrades beyond that — empirically tested in KSP, validated as the right threshold for our 30 Hz simulation. KeplerRails and InterstellarCruise have higher ceilings (10,000× and 100,000×) because their math is analytic and survives arbitrarily high rates. The cap-and-stay policy means a player requesting 1000× on a PhysX vessel sees the effective rate clamp to 5×; the vessel does not auto-transition (deferred to a future commit; see DECISIONS).

**3. Halt surfacing (event bus).** `WarpController` raises `OnRateChanged` and `OnWarpHalted` events; UI elements, audio cues, and Mission Control subscribe independently. This matches the existing `SimTickController.TickAdvanced` pattern — loose coupling, multiple subscribers, no controller knowledge of how the halt is surfaced. Halt registration accepts `WarpHaltInfo` from any source: predictor-driven (atmospheric / surface / SOI), target-tick-reached (internal), manual (UI pause command). The `WarpHaltReason` enum lives in SimTick (not Vessels) as a parallel taxonomy to `TransitionTriggerReason`. This preserves the asmdef direction (Vessels → SimTick → Coordinates); `VesselEventPredictionDriver` translates from predictor-side TriggerReason to halt-side HaltReason at the call site.

**4. Routine-supply gating (`IVessel.IsRoutineSupply`).** Three predictors register halts at the imminent-tick window: atmospheric entry, surface impact, SOI crossing. The first two halt unconditionally — they're terminal events that always warrant the player's attention. SOI crossing is conditional: routine supply runs (Earth-to-Moon cargo loops, etc.) cross SOIs repeatedly and predictably; halting on each crossing is noise. The boolean flag is the minimum-viable classification — extensible later when more vessel-type distinctions emerge (combat, military, science, exploration), but the boolean already captures the load-bearing distinction for v1.

**5. UI scaffolding (`WarpUIController`).** Debug UI for Phase 1 verification — exercises the entire public surface of `WarpController` so testers can manually verify rate changes, pause/resume, target-tick advancement, and the halt-event flow from predictors. Not the production Mission Control UI; that work lands in Phase 5+ with its own polished controller. The lifecycle pattern (Awake for field validation; Start for singleton-dependency check; OnDestroy for paired teardown) sets the precedent for future UI controllers that depend on a singleton initialized by another MonoBehaviour.

## Design decisions

See `docs/DECISIONS.md` "Time-warp controller architecture (commit 048)" for the full multi-part design (six sub-decisions a–f, four alternatives rejected) and the Stage 5 amendment reflecting the cap-and-stay reshape. The headline decisions:

- **Rational rate representation** (`WarpRate` struct) — chosen for determinism. Float multipliers drift over long warp sessions.
- **Cap-and-stay policy for PhysX vessels at 5×** — replaces the pre-commit-048 1× hard cap from `SimTickWarpController`. Force-transition deferred to a future commit; `WarpRateForcedRails` enum value held as infrastructure.
- **Event-bus halt surfacing** (`Action<WarpRate>` / `Action<WarpHaltInfo>`) — loose coupling, multiple-subscriber-friendly, matches existing `TickAdvanced` pattern.
- **`IsRoutineSupply` as single boolean flag** — extensible to richer classification later; minimum-viable for v1.
- **`WarpHaltReason` as parallel-taxonomy enum in SimTick** — preserves Vessels → SimTick asmdef direction; predictor-side mapping translates `TransitionTriggerReason` → `WarpHaltReason` at the registration call site.
- **Integer-only continuous mode in v1** — rational infrastructure ready for fractional rates; ships integer for shipping focus.
- **TMP precedent** — first TMP usage sets pattern for future UI; legacy `UnityEngine.UI.Text` in `TestVesselDriver` and `TestShiftDriver` can migrate opportunistically when touched.

## Lessons

**1. Reconnaissance catches what specs miss.** Stage 2 reconnaissance discovered the existing `SimTickWarpController` with float-multiplier 1× cap — the original Stage 1 spec assumed greenfield implementation. Replacing rather than extending the existing controller was the right move, but the spec didn't know about it. Stage 3 reconnaissance similarly identified that imminent-tick gating was needed (predictors return the same tick repeatedly until the event arrives; without gating, halt would fire on every TickAdvanced). Stage 4 Phase A reconnaissance flagged the Canvas Panel sibling-ordering issue (Panel intercepted clicks via `Raycast Target=true`). Stage 5 recon caught the implementation reality (`ComputeAnalyticIterations` returns 1 during halt, not 0 — the user's spec said "returns 0"; the floor pattern is the right design and the spec was wrong). The pattern: write the spec, then read the code before writing the tests / building the change. Every stage of commit 048 saved blast radius by doing this.

**2. The asmdef-direction-preserving pattern.** `WarpController` lives in `SimTick` and needs to express halt reasons. But SimTick can't reference `TransitionTriggerReason` from Vessels — Vessels references SimTick (for `SimTickController`, `IActiveVessel`, etc.), so the reverse reference would close an asmdef cycle. The solution: parallel-taxonomy `WarpHaltReason` enum in SimTick, with the mapping function in Vessels (`VesselEventPredictionDriver.RegisterHalt`) translating predictor TriggerReason → HaltReason at the registration call site. This pattern will recur whenever a SimTick-side system needs to express Vessel-domain concepts — the predictor work in commits 045–047 already established the inverse direction (Vessels reading SimTick types); commit 048 establishes the asmdef-symmetric mirror.

**3. MonoBehaviour lifecycle race conditions.** `WarpUIController.Awake` originally checked `WarpController.Instance`. Unity does NOT guarantee an order among Awake() calls between MonoBehaviours at the same execution-order priority — and in `TestVessels.unity` the UI controller's Awake won the race, so `Instance` was always null and the defensive `DisableAllInteractables()` fired. Fix: move singleton-dependency check + event subscription from Awake / OnEnable to Start, which Unity guarantees runs after ALL Awakes complete. Paired teardown moved from OnDisable to OnDestroy (debug UI not toggled at runtime — Start / OnDestroy is the right pattern for the use case; OnEnable / OnDisable would be required only for UI that gets toggled in gameplay). This pattern applies to any UI controller that depends on a singleton initialized by another MonoBehaviour and is now documented in `WarpUIController`'s XML doc for future readers.

**4. Operational commits and architectural commits intermix naturally.** Stage 1 absorbed two pre-approved operational cleanups (the `TransitionTriggerReason` split for atmospheric vs surface predictor labels; the `IsRoutineSupply` field addition). Both were independent operational tasks but both were on the critical path for Stage 3's halt-gating logic, and folding them into Stage 1 was cheaper than separate commits — the schema-version bump happens once instead of three times, save-file migration only needs to handle one version step. The discipline: when pre-approved cleanups align with critical-path commit scope, fold them in.

**5. Multi-stage commits with verification gates between stages manage blast radius.** Five stages of commit 048 each landed with green tests before proceeding to the next. When mid-stage reconnaissance revealed problems (Stage 2's existing `SimTickWarpController` requiring an Option-Y reshape; Stage 4's MonoBehaviour lifecycle race requiring an Option-b.2 refactor; Stage 4's Panel sibling-ordering bug discovered during Play-mode verification), the stop-and-surface discipline saved the commit from going wrong. The five-stage cadence trades a small overhead per stage (extra verification step, extra commit message) against the much larger cost of debugging a single fail-everything mega-commit.

**6. Sandbox-vs-host divergence remains the recurring failure pattern.** Workflow rule 6 (host is canonical) caught false alarms throughout Stage 2 (sandbox showed Stage 1 changes uncommitted when they were committed in `f9200ac`; sandbox showed test files apparently truncated mid-comment when they were intact host-side) and Stage 4 (computer-use screenshots showed Unity in a stale layout while host-side the Editor was already in the new layout). Cowork's sandbox view of git state and file state can drift from actual host state — always verify via host-side PowerShell (`git status` / `git log` / `Read` tool against the Windows-authoritative path) before treating sandbox observations as ground truth.

**7. Scene editing risk is real.** Stage 4 Phase B initially failed end-to-end Play verification because the manual scene build produced two Canvas GameObjects (accidental duplication during the build), and later because the Panel was a sibling of the discrete buttons rather than their backing layer (Panel rendered ON TOP of the buttons, intercepting clicks via `Raycast Target=true`). Computer-use cleanup recovered both cleanly: consolidating to one Canvas, moving Panel to first-child position. For non-trivial scene work (UI hierarchies, multiple GameObjects with cross-references), computer-use driving the Unity Editor is safer than direct YAML editing of the `.unity` file — the Editor enforces invariants the YAML doesn't. For simple scene work (drop one prefab, set one Inspector field), manual editing builds Unity intuition and stays fast.

**8. TMP precedent decision matters more than expected.** Commit 048 is the first TMP usage in the codebase. The decision to standardize on TMP for future UI work (rather than the legacy `UnityEngine.UI.Text` used in `TestVesselDriver` / `TestShiftDriver`) was a 3-line addition to `WarpUIController`'s XML doc — but it sets the pattern for every future UI controller. The TMP Examples & Extras package adds 200+ files of font samples and bloat; that was excluded from the import (only TMP Essentials imported). Worth documenting the decision in code (XML doc on `WarpUIController`) so future readers see the reasoning at the point of use, not just buried in a commit artifact.
